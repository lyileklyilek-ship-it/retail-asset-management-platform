# Retail Asset Management Platform
## Платформа управления торговыми активами

---

## О проекте

Платформа для управления торговым оборудованием и промо-материалами в федеральной сети кофеен. Решение построено на базе Oracle CRM с интеграцией аналитики в Power BI.

**Тип продукта:** Гибридное решение (Веб-админка + Мобильное приложение) с офлайн-синхронизацией данных.

---

## Моя роль

**Функциональный аналитик**

В рамках проекта я выполняла:
- Анализ и оптимизацию функциональности модуля управления оборудованием
- Доработку UI/UX веб-интерфейса Oracle CRM (активация кнопок, улучшение навигации)
- Исправление критических багов (создание, деактивация оборудования)
- Формирование требований к мобильному приложению для полевых сотрудников
- Интеграцию с Power BI для аналитических отчетов

## Ключевые особенности

| Особенность | Описание |
|:---|:---|
| Офлайн-режим | Мобильное приложение работает без интернета, данные сохраняются локально |
| Автосинхронизация | При появлении соединения данные автоматически выгружаются на сервер |
| Ролевая модель | Три роли: полевой сотрудник, супервайзер, аналитик |
| Управление оборудованием | Полный жизненный цикл: создание, установка, перемещение, списание |
| Фотоотчеты | Фиксация состояния оборудования с возможностью сравнения "было/стало" |
| Аналитика | Интеграция с Power BI для глубинного анализа данных |

## Ролевая модель

| Роль | Описание | Платформа |
|:---|:---|:---|
| Полевой сотрудник | Выезжает в торговые точки, проверяет наличие оборудования, делает фотоотчеты | Мобильное приложение |
| Супервайзер | Планирует маршруты, проверяет качество работы, утверждает отчеты | Веб-админка (Oracle CRM) |
| Аналитик | Работает с агрегированными данными, отслеживает KPI | Веб-админка + Power BI |

---

## Архитектура решения

**Компоненты системы:**

1. Мобильное приложение — работает офлайн, данные сохраняются локально
2. Oracle CRM (Веб-админка) — управление оборудованием и задачами
3. Power BI — аналитика и дашборды

**Принцип работы:**
Мобильное приложение работает в офлайн-режиме, сохраняя все введенные данные локально на устройстве. При появлении интернет-соединения происходит автоматическая синхронизация с сервером. Веб-админка (Oracle CRM) отображает актуальную информацию и позволяет управлять задачами и оборудованием. Аналитические данные агрегируются и визуализируются в Power BI.

## Описание экранов

### Экран 1: Список оборудования

**Назначение:** Главный экран для управления всеми единицами оборудования. Предоставляет полный обзор всех активов, их статусов и местоположения.

**Пользователи:** Супервайзеры, Аналитики

**Визуальные элементы:**
- Таблица с колонками: ID, Тип, Статус, Торговая точка, Дата установки
- Панель фильтров (по статусу, типу, региону)
- Поисковая строка
- Кнопки действий над таблицей
- Цветовая индикация статусов

**Кнопки действий:**

| Кнопка | Функционал | Условия доступности |
|:---|:---|:---|
| New | Открывает форму создания новой единицы оборудования | Всегда активна |
| Install | Меняет статус на "Установлено" и привязывает к торговой точке | Для оборудования со статусом "На складе" |
| Inactivate | Меняет статус на "Неактивно" (списание) | Для активного оборудования |
| Close Activity | Завершает текущую активность, связанную с оборудованием | Когда есть открытая активность |
| Transfer | Открывает форму перемещения оборудования в другую точку | Для установленного оборудования |

---

### Экран 2: Создание оборудования

**Назначение:** Добавление новой единицы оборудования в систему.

**Вызов:** При нажатии кнопки New на экране списка

**Поля формы:**

| Поле | Тип | Обязательное | Описание |
|:---|:---|:---|:---|
| Тип оборудования | Выпадающий список | Да | Выбор из каталога |
| Серийный номер | Текстовое поле | Нет | Заводской номер |
| Дата поставки | Календарь | Нет | Дата поступления на склад |
| Стоимость | Числовое поле | Нет | Закупочная стоимость |
| Текущее расположение | Выпадающий список | Да | Склад или торговая точка |
| Примечания | Многострочное поле | Нет | Дополнительная информация |

---

### Экран 3: Детальная карточка оборудования

**Назначение:** Просмотр полной информации о конкретной единице оборудования и истории операций.

**Вкладки:**

| Вкладка | Содержимое |
|:---|:---|
| Основная информация | Все поля из формы создания + фото |
| История установок | Где и когда было установлено, кто устанавливал |
| История статусов | Хронология всех изменений статуса |
| Активности | Все задачи, связанные с оборудованием |
| Фотоотчеты | Все фотографии по данному оборудованию |

---

### Экран 4: Мобильное приложение — Список заданий

**Назначение:** Просмотр заданий, назначенных полевому сотруднику.

**Пользователи:** Полевые сотрудники

**Карточка задания содержит:**
- Адрес торговой точки
- Тип задания (установка, проверка, замена)
- Список оборудования
- Приоритет (цветовая индикация)
- Статус (новое, в работе, выполнено, просрочено)

---

### Экран 5: Мобильное приложение — Выполнение задания

**Назначение:** Пошаговое выполнение задания с фиксацией результатов.

**Чек-лист:**
- Пункты с чекбоксами для проверки состояния оборудования
- Возможность отметить "Не применимо"

**Фотофиксация:**
- При нажатии на камеру открывается камера устройства
- Фото сохраняется и привязывается к заданию
- Для заданий на замену — два фото: "было" и "стало"

---

### Экран 6: Дашборд аналитики (Power BI)

**Назначение:** Визуализация ключевых метрик для принятия управленческих решений.

**KPI (ключевые показатели):**

| Показатель | Описание |
|:---|:---|
| Всего оборудования | Общее количество активов в системе |
| Установлено | Процент оборудования, установленного в точках |
| Заданий выполнено | Процент выполнения от плана |
| Активных сотрудников | Количество работающих сотрудников |

**Графики:**
- Динамика установок (линейный график)
- Распределение по статусам (круговая диаграмма)
- Топ точек по оборудованию
- Геораспределение (карта)

## Пользовательские сценарии

### Сценарий 1: Полевой сотрудник выполняет задание

1. Открывает приложение → загрузка списка заданий
2. Нажимает на задание → открывается карточка с деталями
3. Нажимает "Начать" → фиксация геолокации, открытие чек-листа
4. Отмечает пункты чек-листа → прогресс-бар заполняется
5. Делает фото оборудования → фото сохраняется локально
6. Нажимает "Завершить" → задание переходит в статус "Выполнено"
7. Подключается к интернету → автоматическая синхронизация с сервером

---

### Сценарий 2: Супервайзер создает задание

1. Открывает список оборудования → таблица с актуальными данными
2. Нажимает New → открывается форма создания
3. Заполняет поля, сохраняет → оборудование появляется в списке
4. Нажимает Install → открывается выбор торговой точки
5. Выбирает точку, дату, сотрудника → создается задание
6. Проверяет раздел "Задания" → задание отображается в списке активных

---

### Сценарий 3: Деактивация оборудования (до и после доработок)

| Шаг | Действие | До доработок | После доработок |
|:---|:---|:---|:---|
| 1 | Находит оборудование | Успешно | Успешно |
| 2 | Нажимает Inactivate | Ошибка: кнопка неактивна или диалог пустой | Открывается диалог подтверждения |
| 3 | Подтверждает деактивацию | Ошибка сервера | Запрос проходит успешно |
| 4 | Указывает причину | Поле отсутствует | Поле "Причина списания" обязательно |
| 5 | Получает обратную связь | Непонятное сообщение | "Оборудование деактивировано" |

## Мои доработки

### 1. Активация кнопок

**Проблема:** Кнопки New, Install, Inactivate, Close Activity, Transfer были неактивны или работали некорректно. Пользователи вынуждены были использовать сложное контекстное меню.

**Решение:** Активировала кнопки, настроила условия их доступности в зависимости от статуса оборудования.

**Результат:** Интерфейс стал интуитивно понятным, время выполнения операций сократилось.

---

### 2. Исправление создания оборудования

**Проблема:** При нажатии New открывалась форма выбора продукта, но список продуктов был пуст. Пользователь не мог создать оборудование.

**Решение:** Настроила корректное отображение каталога продуктов в форме создания.

**Результат:** Операция создания оборудования стала доступна.

---

### 3. Исправление деактивации оборудования

**Проблема:** При попытке деактивировать оборудование возникала ошибка. Пользователь не мог завершить жизненный цикл актива.

**Решение:** Исправила логику серверного запроса, добавила обязательное поле "Причина списания", настроила корректную смену статуса.

**Результат:** Деактивация оборудования работает корректно, добавлена понятная обратная связь.

---

### 4. Улучшение юзабилити

**Проблема:** Англоязычный интерфейс был более удобен, чем локализованный. Кнопки и элементы управления отличались.

**Решение:** Привела локализованную версию к функциональности англоязычной, добавила недостающие кнопки.

**Результат:** Локализованная версия интерфейса стала такой же удобной и функциональной, как английская.
